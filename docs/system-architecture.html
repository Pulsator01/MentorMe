MentorMe Architecture

Frontend and Backend System Architecture

This rendered document describes the current MentorMe system as implemented in the repository: the React frontend, the Fastify backend, the request lifecycle, the planned persistence model, and the key architectural gaps between the current demo runtime and the intended production shape.

Frontend runtime React + Vite

Role-based SPA with a shared state provider and optional backend hydration.

Backend runtime Fastify

Service-layer API using explicit interfaces for repository, email, storage, and queue.

Persistence target Prisma + PostgreSQL

Schema exists now; active server runtime still uses an in-memory repository.

Async model Outbox scaffold

Worker is present but not yet processing events beyond startup inspection.

Frontend

Frontend Architecture

Responsibilities

  • Render a role-based single-page application.
  • Separate founder, student, and CFE workflows into distinct routes.
  • Keep business state in a single provider.
  • Switch between local seeded mode and API-backed mode.

Key frontend patterns

  • Single shared shell via SidebarLayout.
  • State orchestration through AppStateProvider.
  • Manual fetch client instead of a query library.
  • Page-specific workflow components rather than one overloaded dashboard.

Backend

Backend Architecture

Responsibilities

  • Authenticate local users via magic-link flow.
  • Authorize role and venture access.
  • Enforce mentor request lifecycle rules.
  • Manage mentors, artifacts, outreach, scheduling, and feedback.
  • Emit audit and outbox records.

Current runtime adapters

  • inMemoryRepository for active storage.
  • stubEmailGateway for magic-link and outreach capture.
  • stubStorageService for presigned uploads.
  • inlineQueuePublisher for async event capture.

Operational Notes

Current Runtime Modes, Gaps, And Next Steps

Runtime modes

  • npm run dev: frontend-only seeded demo.
  • npm run dev:full: Vite + Fastify API + worker scaffold.
  • VITE_API_BASE_URL enables API hydration in the frontend provider.

Architectural gaps

  • Backend persistence is still ephemeral.
  • SSE exists but is not consumed by the frontend.
  • Worker is scaffolded, not operational.
  • Real email, storage, and queue infrastructure are not wired in yet.
The current repository already has strong interface boundaries. The main missing step is swapping demo adapters for durable and asynchronous infrastructure without changing the API or UI contracts.
